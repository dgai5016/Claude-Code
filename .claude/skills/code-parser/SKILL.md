我是**前端与AI领域的初学者**，请你用最简单、最通俗、最细致的方式帮我解析下面这段源码。
我正在学习 Claude-Code 源码，需要你严格遵守以下规则：

--------------------------------------------------

【1】文件整体解释
- 放在【代码最顶部】
- 用初学者能听懂的话说明：文件作用、模块定位、调用关系、设计意图
- 在文件最底部加上一行引导："👇 详细架构设计图请见文件底部【架构设计附录】"
--------------------------------------------------

【2】导入注释 —— 必须写明目的
- 每个 import 语句上方都要有注释
- 注释必须包含两部分：【导入了什么】+【在本文件中用来干什么】
- 格式示例：
  // 导入 Commander.js 命令行框架，用于定义和解析 claude 的所有命令行参数和子命令
  import { Command } from '@commander-js/extra-typings';
- 反面示例（禁止）：
  // 导入 Commander.js              ← 只说了是什么，没说用来干嘛
  // 从 commander 导入 Command       ← 只说了导入什么，没说目的
- 对于类型导入（import type），注释中说明"用于类型标注——xxx"
- 对于动态导入（import() / require()），注释中说明为什么要延迟加载
--------------------------------------------------

【3】代码块解释
- 每个重要代码块【上方】写一段自然语言解释
- 用大白话讲：这段代码干什么、为什么这么写、核心逻辑
- 避免专业术语堆砌，尽量通俗
--------------------------------------------------

【4】行级别注释
- 所有行都增加注释
- 在【代码行正上方】添加注释
- 每行注释通俗易懂
- 不修改原代码
- 不删除原注释
--------------------------------------------------

【5】架构设计附录（放在代码最底部）
- 放在【所有原始代码之后】，用注释格式包裹（不影响代码执行）
- 包含以下 5 个子部分，每个子部分用分隔线隔开
- 风格：简洁直接，标注行号，不搞文学比喻，不搞花哨排版

  【5.1】文件结构总览
  - 列出文件的几个大块，每块写：编号、名称、行号范围、一句话说明
  - 示例格式：
    1. 导入区（行1~676）导入所有依赖，前3行启动并行任务
    2. 辅助函数（行693~1500）main/run 需要的工具函数
  - 附录开头给出一句阅读建议

  【5.2】执行流程
  - 按执行顺序列出步骤，每步标注代码行号
  - 有分支的地方用缩进表示（如交互式/无头两条路径）
  - 示例格式：
    1. 行9~20    启动3个并行任务（MDM/钥匙串/打点）
    2. 行1503    main() 入口
      ├── 行1809  判断交互/非交互
      └── 行2070  调用 run()
  - 不要写比喻！直接说做了什么、在哪个行号

  【5.3】数据流
  - 用缩进箭头链展示一个请求从输入到输出的路径
  - 每步标注对应的处理函数和数据类型
  - 示例格式：
    process.argv
      → Commander 解析 → { print, prompt, model }
        → getInitialSettings() → Config
          → initializeToolPermissionContext() → PermissionContext
  - 末尾列出贯穿全程的关键对象及其字段
--------------------------------------------------

【6】绝对禁止
- 不修改原代码任何字符
- 不删除原注释
- 不改变结构
- 不重构
- 不使用过于高深的专业术语
- 不换行长代码行（原文件中的长行必须保持原样，哪怕一行有几百个字符）
- 不对代码做任何格式化或美化
--------------------------------------------------

【7】输出格式
1. 顶部文件说明（初学者友好版）
2. 导入语句上方 → 写明【导入了什么】+【用来干什么】
3. 代码块上方 → 通俗块解释
4. 关键行上方 → 简单易懂注释
5. 所有行上方 → 简单易懂注释
6. 底部 → 架构设计附录（5个子部分：路线图、执行流程、数据流动、设计套路、关键决策）
7. 输出直接保存为原来的文件格式
8. 附录用注释格式包裹，不影响代码执行
--------------------------------------------------

【8】注意点
1. 如果源码行数太多，可以自动开启多Agent来分块处理
2. 多Agent处理时，每个Agent只负责在自己分到的行范围内添加注释，绝不修改原始代码行（包括不换行、不格式化）
3. 多Agent处理时，架构附录由主Agent在所有块注释完成后统一生成（附录需要全局视角）
--------------------------------------------------

【9】验证
1. 最后要验证一下，不要有任何删除的代码行
2. 验证代码行注释，确保每行都有通俗易懂的注释
3. 验证导入注释都写了目的（不只是"导入了什么"）
4. 验证架构附录的 ASCII 图表在各行宽度上对齐正确
5. 验证附录中的函数名与源码中实际函数名一致
6. 验证原代码行没有被换行或格式化（长行必须保持原样）
